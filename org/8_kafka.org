* Examples of modern data architecture

- *Lambda architecture* (2011): handles big data in real-time.
  1. The batch layer aims at perfect accuracy (immutable).
  2. The speed layer is not perfect but fast (low latency).
  3. The service layer unifies batch and speed layer views.
  #+attr_html: :width 400px :float nil:
  [[../img/lambda.png]]

- Lambda architecture ensures human-fault tolerance but is more
  complex with dual data pipelines. Examples: Hadoop, Apache Spark
  (speed layer), Apache Cassandra (NoSQL database).

- *Kappa architecture*:
  1. Solely relies on real-time streaming (batch = special case of
     streams, replaying historical logs)
  2. Only a speed layer - always fast.
  3. No serving layer: Outputs written directly to views from the
     stream processor. No merging required.
  #+attr_html: :width 400px :float nil:
  [[../img/kappa.png]]

- Kappa architecture simplifies Lambda architecture: it's ideal for
  stream-native environments but less suited if batch recomputation is
  frequent or if storage costs for long logs prohibitive. Examples:
  Apache Kafka.

- *Use cases:*

  1) Lambda: Fraud detection in finance analyzes systems in real-time
     alongside historical patterns to detect anomalies. Batch layers
     used for audits, speed layers for immediate alerts.

  2) Kappa: Uber analyses ride data instantly using a unified Apache
     Kafka (stream storage and connect) and Flink (stream processing).

- In addition (outside of our scope) there are orchestration tools
  that schedule and manage ETL pipelines (like Apache Airflow) and
  DAGs (Directed Acyclic Graphs) - two data engineering concepts.

- *Issues:* The more moving parts are present, the more difficult data
  governance and operations become (leading to DevOps and MLOps
  methodologies). The abstraction only simplifies the interface, not
  the implementation, and generates more and more *metadata*.

* Why Kafka Is an Excellent Illustration of Computer Architecture Concepts

** Built Directly on Core Architecture Principles

Kafka’s design maps cleanly to fundamental architecture ideas:
append-only logs, sequential disk writes, and log-structured storage.
You can see how low-level I/O patterns shape high-level system
behavior.

** Exposes Hardware Behavior Instead of Hiding It

Kafka does not abstract away the OS or hardware. It leverages:
- the page cache,
- kernel read-ahead,
- filesystem append semantics,
- zero-copy transfer.
These make hardware-level effects visible and measurable.

** Demonstrates Parallelism Clearly (Partitions = Shards)

Kafka partitions illustrate data parallelism:
- multiple partitions on one machine → multi-core parallelism,
- partitions spread across machines → horizontal scaling.
You can see how partitioning maps to architectural parallelism.

** Shows the Performance Difference Between Sequential and Random I/O

Kafka’s high throughput comes from sequential disk writes.  The
contrast to random I/O demonstrates:
- disk seek behavior,
- cache locality,
- OS-level buffering.
Architecture concepts become directly tied to real-world performance.

** Makes Durability and Fault Tolerance Concrete

Kafka replication and commit logs illustrate:
- redundancy,
- failsafe disk writes,
- synchronous vs. asynchronous replication,
- RAID-like trade-offs.
These map directly onto storage architecture and reliability models.

** Operationalizes Memory and Concurrency Models

Kafka’s ordering and visibility guarantees depend on:
- memory consistency,
- write barriers,
- concurrency control,
- atomic commits.
Yo see that distributed guarantees are rooted in CPU/OS models.

** Industry-Relevant Example of Architecture in Action

Kafka is widely used in data engineering, microservices, and stream
processing. It provides a practical, high-impact way to show why CPU,
memory, I/O, and OS concepts still matter in modern distributed
systems.

* Distributed computing challenges

- *Append-only logs* (sequential disk I/O, log-structured storage)
- *Partitioning and sharding* (data parallelism)
- *Replication* (fault tolerance, redundancy)
- *Page cache use / OS-level caching*
- *Sequential vs. random I/O performance*
- *Producer–consumer buffering*
- *Backpressure and flow control*
- *Distributed consensus* (controller election, ISR management)
- *Message ordering and memory models*
- *High-throughput networking* (zero-copy, batching)

* How does Apache Kafka map onto the von Neumann architecture?

1. CPU
   - Role: Executes broker processes, handles compression, network,
     replication; I/O-bound, not intensive.
   - Optimization: Multi-core (8-12/broker); low overhead with KRaft.

2. RAM
   - Role: OS page cache for buffering/caching messages; JVM heap
     ~4-6GB (Kafka is written in Java), rest for cache.
   - Optional: 32-64GB/broker; maximizes reads from memory, avoids
     garbage collection pauses.

3. I/O Bus
   - Role: Connects to disks (sequential writes/reads) & network
     (replication, clients).
   - Opt: Multiple SATA HDDs (JBOD); 10GbE+ network; partitioning for
     load balance.

* Apache Kafka Theory

- Apache Kafka functions as a distributed streaming platform that
  treats data as immutable event logs known as topics, promoting an
  event-first approach over traditional databases.

- It supports scalable and fault-tolerant real-time data processing
  through decoupled producers that write data and consumers that read
  it.

- Core elements include brokers for storage, partitions for
  parallelism, replication for durability, and offsets for tracking
  consumption progress.

- The system is optimized for high-throughput workloads, ensuring
  minimal downtime in handling large-scale streaming applications.

- Sources: This section is largely based on DataCamp's "Introduction
  to Apache Kafka" (by Mike Metzger) and Udemy's "Apache Kafka Series"
  (Stephane Maarek, conduktor.io), with occasional help from GenAI.

** Kafka topics
#+attr_html: :width 600px :float nil:
[[../img/topics.png]]

- Topics act as unconstrained streams of data similar to database
  tables, accommodating various message formats like logs or GPS
  updates.

- They are partitioned for ordered, immutable message sequences, with
  configurable retention periods such as one week.

- Offsets serve as partition-specific pointers, guaranteeing order
  only within partitions unless keys direct distribution.

- This enables multiple independent services to consume the same
  stream for applications like real-time analytics.

** Kafka Producers

- Producers handle writing data to partitioned topics, automatically
  selecting brokers and recovering from failures.

- Messages can include keys for consistent hashing via murmur2 to
  ensure ordering in the same partition, or null keys for round-robin
  load balancing.

- Each message contains keys, values, timestamps, headers, and
  compression, serialized into bytes using formats like Avro or
  Protobuf.

- Serialization maintains compatibility, allowing deterministic
  routing for fields like truck IDs in GPS tracking.

- How a Kafka message looks like when created by the producer before
  being sent to storage:
  #+attr_html: :width 400px :float nil:
  [[../img/message.png]]

- Messages have to be serialized to transform the data into bytes. The
  serializers are used on the value and the key.

- Example: Message serialization for a {key,value} pair.
  #+attr_html: :width 400px :float nil:
  [[../img/serializer.png]]

- Common serializers: String (JSON), int, float, avro, protobuf
  #+begin_quote
  Apache Avro is a serialization framework developed for Apache
  Hadoop. Protocol Buffers (Protobuf) is a language- and
  platform-neutral serialization format developed by Google. It is
  faster and smaller than XML or JSON. Serialization involves schema
  handling between producer and consumer - i.e. adding fields without
  breaking existing applications.
  #+end_quote

** Kafka Consumers
#+attr_html: :width 600px :float nil:
[[../img/consumer.png]]

- Consumers pull ordered data from topics via brokers, recovering from
  failures and reading from low to high offsets per partition.

- No cross-partition ordering is assured, focusing reliability on
  sequential consumption within partitions.

- Deserialization converts byte messages back to usable formats,
  requiring matching configurations with producer serializers.

- This ensures efficient stream processing while preserving topic
  lifecycle compatibility.


- The consumer needs to know in advance what the expected data formats
  are to deserialize - it needs to be configured accordingly.
  #+attr_html: :width 400px :float nil:
  [[../img/deserializer.png]]

- The serializer (producer) / deserialization (consumer) type must not
  change during a topic lifecycle (instead create a new topic).

** Consumer groups and consumer offsets

- Groups enable collaborative reading where consumers handle exclusive
  partitions, with extras as inactive standbys for scalability.

- Multiple groups can consume the same topic independently, with
  offsets stored in =__consumer_offsets= for progress tracking.

- Periodic offset commits allow recovery after consumer failures,
  supporting delivery semantics like at least once or exactly once.

- Idempotence is advised to manage duplicates, balancing reliability
  with performance in processing.

- Consumers in an application read data as a group - each from
  exclusive partitions.
  #+attr_html: :width 600px :float nil:
  [[../img/consumer_groups.png]]

- If you have more consumers than partitions, some consumers will be
  inactive (stand-by).
  #+attr_html: :width 600px :float nil:
  [[../img/inactive_consumer.png]]

- You can have multiple consumer groups on the same topic.
  #+attr_html: :width 600px :float nil:
  [[../img/consumer_topics.png]]

- As a table:
  | Concept        | Relationship To…                 | Cardinality         |
  |----------------+----------------------------------+---------------------|
  | Topic          | → Partitions                     | 1 → N               |
  | Partition      | → Topic                          | N → 1               |
  | Partition      | → Consumer (in consumer group)   | 1 → 1               |
  | Consumer       | → Partitions (in consumer group) | 1 → N               |
  | Consumer       | → Application                    | N → 1               |
  | Application    | → Consumers                      | 1 → N               |
  | Topic          | → Consumer Groups                | 1 → N               |
  | Consumer Group | → Partitions                     | N → N (constrained) |

- In prose:
  1) A topic is split into multiple partitions.
  2) Each partition belongs to exactly one topic.
  3) A partition is consumed by exactly one consumer within a group.
  4) A consumer may read from multiple partitions.
  5) Many consumers can belong to one application instance.
  6) Applications scale by running many consumers.
  7) Many consumer groups can read the same topic independently.
  8) Partitions are distributed across consumers.
  9) The number of consumers is smaller than the number of partitions.
  10) If you have more consumers than partitions some are inactive.

- When a consumer in a group has processed data received from Kafka,
  it should be periodically committing the offsets.
  #+attr_html: :width 600px :float nil:
  [[../img/consumer_offsets.png]]

- If a consumer "dies" (digitally) and comes back to life, it can read
  back from where it left off b/c of the committed consumer offsets.

- Delivery semantics for consumers (for manual commit):

  1) At least once: commit offsets after message is processed. If it
     goes wrong, message will be read again (with idempotence:
     repeated processing does no harm).
  2) At most once: offsets are committed as soon as messages are
     received.
  3) Exactly once: for Kafka-to-Kafka workflows (using the
     transactional API, Kafka Streams API).

- Explanation:
  #+begin_quote
  These "delivery semantics" describe what guarantees you get
  depending on when you commit an offset (a pointer to where you are
  in partition at the time):
  1) *At least once* means commit *after* processing:
     1. Read message
     2. Process message
     3. Commit offset
     -----
     - *Guarantee:* You will never miss a message.
     - *Risk:* If consumer crashes after processing but before commit,
       the offset is not updated and the message is delivered again.
     - *Outcome*: Duplicate processing is possible (got to be idempotent
       to be safe).
     - *Use*: Most applications (safe default); logging, analytics, DB
       writes.

  2) *At most once* means commit *before* processing:
     1. Receive message
     2. Commit offset
     3. Process message
     -----
     - *Guarantee:* You will never process a message twice.
     - *Risk:* If the consumer crashes after commit but before
       processing, the message is lost forever (Kafka thinks it's
       already processed).
     - *Outcome*: Fast, but unsafe.
     - *Use*: Low-value telemetry, monitoring updates, anything where
       losing some message is acceptable.

  3) *Exactly once* is only guaranteed for Kafka-to-Kafka workflows,
     using transactions.
     1. Message is processed once.
     2. Result is written once.
     3. Consumer's offset is committed atomically with the output.
     -----
     - *Guarantee:* No duplication, no loss. Only within Kafka. Not
       available to general external sinks like databases unless the
       external system itself supports idempotent/transactional
       writes.
     - *Use*: Stream processing pipelines topic->processing->topic

  | Semantics     | When Offset Is Committed      | Duplicate Risk | Loss Risk |
  |---------------+-------------------------------+----------------+-----------|
  | At least once | After processing              | Yes            | No        |
  | At most once  | Before processing             | No             | Yes       |
  | Exactly once  | Transactionally (Kafka↔Kafka) | No             | No        |
  #+end_quote

- Clusters consist of multiple ID-identified brokers hosting topic

  #+attr_html: :width 600px :float nil:

  send data to the broker that leads a partition. Here, 101 leads
  partition 0, and 102 leads partition 1.
  #+attr_html: :width 600px :float nil:

- ISR = In-Synch-Replica:
  #+begin_quote
  Kafka keeps multiple copies (replicas) of each partition for fault
  tolerance. The *In-Sync Replicas* (ISR) are the set of replicas that:

  - have fully caught up with the leader’s log, and
  - are actively receiving new writes with minimal lag.

  Only replicas in the ISR are eligible to take over as leader if the
  current leader fails. If a replica falls too far behind, it is
  removed from the ISR until it catches up again.
  #+end_quote

- Producers can only write to leader broker for a partition, consumers
  by default will read from it.
  #+attr_html: :width 600px :float nil:
  [[../img/leaders.png]]

- Since v2.4, consumers can be configured to read from the closest
  replica, too
  #+attr_html: :width 600px :float nil:
  [[../img/leaders2.png]]

** Producer acknowledgement & topic durability

- Acknowledgments (=acks=) options include 0 for speed with loss risk, 1
  for leader-only with limited loss, and all for full replica
  confirmation with no loss.

- Replication factor N allows tolerance of up to N-1 broker failures
  without data loss.

- This configurable balance supports high durability for critical
  applications.

- Acks enhance topic reliability in varying operational scenarios.

- Producers can choose to receive acknowledgements of data writes:
  | ~acks=0~   | won't wait for acknowledgement       | possible data loss |
  | ~acks=1~   | will wait for leader acknowledgement | limited data loss  |
  | ~acks=all~ | leader + replicas acknowledge        | no data loss       |

- For a topic replication factor = 2, Kafka can lose 2 brokers: for a
  replication factor of N > 1, you can permanently lose up to N-1 brokers
  and still recover your data.

** Zookeeper

- Zookeeper coordinates Kafka by managing broker lists, leader
  elections, metadata storage, and change notifications.

- It uses odd-numbered ensembles for quorum reliability but is phasing
  out in favor of KRaft from Kafka 3.x.

- Full removal is set for version 4.0, with restricted ports for
  security to brokers only.

- This service ensures cluster stability in older Kafka setups.

- Zookeeper operates with an odd number of servers (an ensemble):
  #+attr_html: :width 600px :float nil:
  [[../img/zookeeper.png]]

- All APIs and commands that used Zookeeper are migrated to use Kafka
  instead so that when clusters are migrated, no change is visible.

- Zookeeper is less safe: Zookeeper ports should only be opened to
  allow traffic from Kafka brokers, not Kafka clients.

** Why Older Versions of Kafka Required ZooKeeper

- ZooKeeper enabled leader election and tracking for partitions,
  ensuring single-leader reliability.

- It stored cluster metadata like broker registrations, topics, and
  early ISR details.

- Failure detection via heartbeats allowed rebalancing, with
  consistency preventing split-brain issues.

- As Kafka's external coordination layer, it provided atomic changes
  until replaced by internal KRaft.

- ZooKeeper provided the distributed coordination services that Kafka
  originally did not implement itself. Specifically, ZooKeeper handled:

  1. Leader Election: Kafka partitions need a single leader
     replica. ZooKeeper elected and tracked the leader for every
     partition reliably.

  2. Metadata Storage: ZooKeeper stored cluster-wide metadata:
     - broker registrations
     - topic configurations
     - partition assignments
     - ISR (in-sync replica) membership (early versions)

  3. Failure Detection: ZooKeeper monitored broker heartbeats. When a
     broker died, ZooKeeper informed Kafka so partitions could be
     rebalanced.

  4. Consistency Guarantees: ZooKeeper provided a strongly consistent
     coordination layer that Kafka relied on for:
     - atomic state changes,
     - avoiding split-brain,
     - ensuring only one leader at a time.

  - Summary: Kafka originally lacked built-in coordination
    mechanisms. ZooKeeper served as the external, reliable “source of
    truth” for cluster state, leadership, and failure detection until
    Kafka’s own Raft-based metadata quorum (KRaft) replaced it.

** Kafka KRaft (removing Zookeeper - since 2020)

- KRaft embeds Raft consensus in Kafka for metadata, overcoming
  ZooKeeper's scaling limits beyond 100,000 partitions.

- It supports millions of partitions, simplifying monitoring,
  security, administration, and recovery.

- APIs remain unchanged for seamless migration, introduced in 2020 and
  standard from 3.x.

- ZooKeeper support ends by Kafka 4.0, boosting overall cluster
  performance.

** Summary

- Kafka clusters feature brokers with replicated, partitioned topics,
  leaders, and ISRs for fault tolerance.

- Producers employ keys, round-robin, and acks for ordered, durable
  writes.

- Consumers use groups, offsets, and semantics for reliable reading.

- The shift from ZooKeeper to KRaft enhances coordination in scalable
  streaming architectures.

* Using Kafka Lab 1: The Foundation (Read & Write)

** The Problem: Batch vs. Streaming

*Concept:* Imagine a retail store manager. At 9:00 PM every night, they
download a CSV file of all the day's sales, put it on a USB stick, and
email it to Headquarters.

*Questions:*
- If a credit card was stolen at 10:00 AM and used for fraud, when
  does HQ find out? (Answer: Not until the next day).
- This is called *Batch Processing*. It's like a bucket of water. You
  wait for it to fill up, then you carry it.

*The New Way (Event Streaming):*
- In modern systems (Uber, Netflix, Banks), every mouse click, every
  ride request, and every payment is an *Event*.
- We process these events *continuously*, the millisecond they happen.
- This is *Stream Processing*. It's like a garden hose—the water flows
  constantly.

** The Solution: Apache Kafka

*Definition:* Apache Kafka is a distributed event streaming platform. It
connects things that create data (Producers) to things that use data
(Consumers).

*The "Secret Sauce": The Append-Only Log*
- Why are standard databases (SQL) slow for this?
  #+begin_export ascii
  They have to jump around the hard drive to update rows (Random I/O).
  #+end_export

- Why is Kafka fast?
  #+begin_quote
  - Kafka is an *Append-Only Log*. It only writes to the *end* of the file
    (Sequential I/O).
  - It relies on the physics of the disk: writing in a straight line
    is faster than jumping around.
  #+end_quote

- *Key Kafka architecture concepts:*
  - *Topic:* A category or feed name (like a folder).
  - *Producer:* The app sending data.
  - *Broker:* The server storing the data.
  - *Consumer:* The app reading the data.

** Anatomy of a Message

A message packet contains:
1. A *Key:* (Optional) Used for routing.
2. A *Value:* The actual data (JSON, Text, Binary).
3. A *Timestamp:* When it happened.
4. A *Headers:* Metadata (like email headers).

** The "Router" Logic (Keys)

*The Big Question:* If we have a Topic with 3 Partitions (3 folders),
how does the Producer decide which folder to drop the message into?

*Scenario A: Key = Null*

- Logic: *Round Robin* ("Load Balancing").
- "Message 1 -> Part 0", "Message 2 -> Part 1", "Message 3 -> Part 2".
- *Pros:* Even distribution.
- *Cons:* No ordering guarantee between related messages.

*Scenario B: Key = "Truck-101"*
- Logic: *Hashing*.
- Formula: ~targetPartition = hash(Key) % numPartitions~ (murmur 2)
- *Guarantee:* All messages with the Key "Truck-101" will ALWAYS go to
  the exact same partition.
- *Why?* To keep the GPS history of that specific truck in strict
  chronological order.

Both =truck1= and =truck2= messages went to =Partition 1=. Why?

We created the topic with 3 partitions 0,1,2. The hashing formula
target partition = hash(key) % number of partitions, yielded:
#+begin_example
Key "truck1": Kafka hashed it -> divided by 3 -> Remainder was 1.
Key "truck2": Kafka hashed it -> divided by 3 -> Remainder was 1.
Key "truck3": Kafka hashed it -> divided by 3 -> Remainder was 2.
#+end_example

What does matter is that all =truck1= messages go to 1.

The fact that =truck2= is also in 1 is fine:
- In a real system, you may have 100 partitions and 10,000 trucks.
- Every partition will only hold data for 100 different trucks.
- Partition 1 is the 'apartment' shared by =truck1= and =truck2=.

*** Concept: Hash Collisions & Partition Assignment

**** The Observation (What you saw)

When you ran the "Inspection Consumer" with ~--print-key~, you noticed
that both ~truck1~ and ~truck2~ landed in the same partition (Partition
1), while ~truck3~ landed elsewhere (Partition 2).

#+begin_example
Partition:1      truck1 moving north
Partition:1      truck2 moving south
Partition:1      truck1 stopped
Partition:1      truck2 delivering
Partition:2      truck3 starting engine
#+end_example

**** The Math (Why it happened)

Kafka uses the *Murmur2* hashing algorithm to turn the Key string into
an integer, then uses the *Modulo* operator (%) to assign it to one of
the 3 available partitions (0, 1, 2).

$\text{Target Partition} = \text{abs}(\text{Murmur2}(\text{Key})) \%\text{NumPartitions}$

In our specific case with 3 Partitions:

| Key      | Hash (Approx) |  Abs Value | Math (/ 3)  | Mod (%) | Result      |
|----------+---------------+------------+-------------+---------+-------------|
| "truck1" |   -1032868205 | 1032868205 | 344,289,401 | *1*       | Partition 1 |
| "truck2" |    -243179237 |  243179237 | 81,059,745  | *1*       | Partition 1 |
| "truck3" |    1285324316 | 1285324316 | 428,441,438 | *2*       | Partition 2 |

*The Hash Collision:* Both ~truck1~ and ~truck2~ resulted in a remainder of
*1*. This is a "Hash Collision" in the context of partition
assignment. It is normal and expected behavior.

*** The Analogy (The "House Number" Rule)

Imagine a street with exactly 3 houses (Partition 0,1,2). You are a
mail carrier. You have a rule: take the last digit of the ZIP code,
divide by 3, and the remainder is the house number.

- Letter 1 (ZIP 90210): Ends in 0. 0 / 3 remainder is 0. Deliver to House 0.
- Letter 2 (ZIP 90210): Ends in 0. 0 / 3 remainder is 0. Deliver to House 0.
- Letter 5,000 (ZIP 90210): Ends in 0. 0 / 3 remainder is 0. Deliver to House 0.

You don't need to remember where you put the previous letters. As long
as the ZIP code (Key) stays 90210 and the number of houses
(Partitions) stays 3, the math always yields the same result.

This rule ("stateless hashing") only breaks down if the number of
partitions changes - which is why Kafka does rarely change the number
of partitions on a live topic. If it does, the existing keys jump and
you lose ordering guarantee for that specific moment (then you need to
"rehash").

*** The Takeaway (Ordering Rules)

- *Success:* Even though they share a "room" (Partition 1), ALL ~truck1~
  messages are together.

- *Guarantee:* Kafka guarantees that if you send 1,000 messages for
  ~truck1~, they will *all* go to Partition 1. They will never be split
  across 0 or 2.

- *Strict Ordering:* Because they are in the same partition, the
  consumer is guaranteed to read "moving north" *before* it reads
  "stopped".

We need a topic with 3 partitions to allow 3 consumers to work in parallel.

*** Start the Consumer Group

We will simulate 3 different computers connecting to this topic.
1. Open *3 separate Terminal Tabs* (let's call them C1, C2, and C3).

1. Open a (new) *Producer Tab* (this is your 4th or 5th tab).

3. At the prompt, type messages rapidly (simulate a busy day):
   #+begin_example
   order-101
   order-102
   order-103
   order-104
   order-105
   order-106
   #+end_example

*** Verify Load Balancing

1. Click through your 3 Consumer Tabs (C1, C2, C3).
   
2. You will (probably) see that all messages ended up in C1
   anyway. Kafka was not fooled: It is inefficient to open connections
   to different partitions for tiny packets.

3. *Conclusion*: You have to simulate a real flood.

*** Flood the System with Messages 2

1. Close the manual producer (CTRL + C).

2. Run this simple bash script to fire 50 messages in 1 second:

   #+begin_example sh
   for i in {1..50}; do echo "order-$i"; done | ./bin/kafka-console-producer.sh --topic orders --bootstrap-server localhost:9092 --producer-property batch.size=1
   #+end_example

3. Looking better: the ~describe~ command gives you some partially
   filled partitions, while C3 may still be empty.

   #+begin_example
   GROUP  TOPIC  PARTITION  CURRENT-OFFSET  LOG-END-OFFSET  LAG  CONSUMER-ID                                           HOST       CLIENT-ID
   my-app orders 1          573             573             0    console-consumer-8c60e378-ae82-4d6d-8024-3ab2d55d9316 /127.0.0.1 console-consumer
   my-app orders 2          16              16              0    console-consumer-a7af794a-4fee-41cd-88f0-bdad3ebf56e7 /127.0.0.1 console-consumer
   my-app orders 0          72              72              0    console-consumer-1e8b7ff6-cf78-40bc-9da1-b9a72d9ffdc1 /127.0.0.1 console-consumer
   #+end_example

4. Observation:

   - Look at the PARTITION column: All three partitions (0, 1, 2) now
     have data (offsets > 0).
   - Look at the CONSUMER-ID column: We see three different IDs (e.g.,
     ...9316, ...56e7, ...fdc1).
   - The message counts are uneven (Partition 1 has 573, while
     Partition 2 has only 16) because Partition 1 absorbed the heavy
     traffic from our previous "sticky" test. However, the new traffic
     is finally reaching the other partitions.

5. Conclusion:

   - Success: We have achieved Horizontal Scaling.
   - By forcing the Producer to break its "Sticky" habit (using
     batch.size=1), we proved that Kafka can distribute work to 3
     separate computers simultaneously.
   - If we had 1 million messages now, all 3 consumers would be
     working together to clear the queue 3x faster than a single
     machine could.

** Review I: Topics and Producers

1. On Message Ordering: You have a topic named =truck_gps= with 10
   partitions. You need to read messages in the exact order they were
   created. Does Kafka guarantee that all messages in the =truck_gps=
   topic are ordered by default?

   #+begin_quote
   No. Order is not guaranteed across the entire topic, only within a
   specific partition. Offsets only have meaning within that
   partition.
   #+end_quote

2. On Producer Keys (Load Balancing): If a Producer sends a message
   with =key=null=, how does Kafka decide which partition to write the
   data to?

   #+begin_quote
   If =key=null= then the partitions are written to in round robin
   style, starting from partition 0 to n and wrapping around until all
   messages are written.
   #+end_quote

3. On Producer Keys (Ordering): You want to ensure that all GPS data
   for "Truck ID 101" always ends up in the exact same partition. What
   specific action must the producer take to make this happen?

   #+begin_quote
   Send the truck ID as the key. If =key!=null= then all messages
   sharing that key will map to the exact same partition.
   #+end_quote

4. The "Under the Hood" Math: When a key is provided, the producer
   uses a formula to calculate the target partition: What is the name
   of the hashing algorithm used here?
   #+begin_example
   targetPartition=Math.abs(Utils.murmur2(keyBytes)) % (numPartitions - 1)
   #+end_example
   #+begin_quote
   murmur2 is the default hashing algorithm used here.
   #+end_quote

5. Anatomy of a Message: Besides the actual Key and Value (and the
   Partition+Offset), name at least two other fields that are
   contained inside a Kafka message when it is created.

   #+begin_quote
   Timestamp, compression type, and message headers.
   #+end_quote

** Review II: Consumers and Brokers

1. How many partitions can a Kafka topic have?
   #+begin_quote
   As many as you want; Kafka imposes no fixed limit, but performance
   and manageability set practical limits.
   #+end_quote

2. At which level are offsets relevant - topic, topic-partition?
   #+begin_quote
   Offsets exist at the *topic-partition* level; each partition has its
   own offset sequence.
   #+end_quote

3. What are offsets relevant for - producer or consumer?
   #+begin_quote
   Offsets matter to *consumers*; producers do not track offsets.
   #+end_quote

4. Once sent to a topic, a message can be modified. True or false?
   #+begin_quote
   False; Kafka logs are immutable—messages cannot be altered after
   being written.
   #+end_quote

5. What are brokers identified by?
   #+begin_quote
   Each broker is identified by a unique numeric broker ID.
   #+end_quote

6. What is an alternative name for Kafka brokers?
   #+begin_quote
   "Kafka servers"—the terms are interchangeable.
   #+end_quote

7. A broker contains only a subset of the topics and the
   partitions. True or false?
   #+begin_quote
   True; brokers host only the partitions assigned to them, not the
   entire cluster’s data.
   #+end_quote

8. How many brokers will a partition live on if the replication factor is 3?
   #+begin_quote
   Exactly 3 brokers: one leader and two followers.
   #+end_quote

9. If a topic has a replication factor of 3, what maximum number of
   brokers can be stopped without impacting the topic availability?
   #+begin_quote
   You can lose 2 brokers and still have one in-sync replica
   available.
   #+end_quote

10. How many leaders and how many replicas can a partition have?
    #+begin_quote
    One leader per partition; replicas = replication factor (including
    the leader).
    #+end_quote

11. What is a leader?
    #+begin_quote
    The broker replica responsible for handling all reads and writes
    for a partition.
    #+end_quote

12. What does ISR mean?
    #+begin_quote
    In-Sync Replicas: the set of replicas fully caught up with the
    leader’s log.
    #+end_quote

13. What must a producer provide to a Kafka client to send data to a
    topic?
    #+begin_quote
    The bootstrap server list (brokers) and the target topic name.
    #+end_quote

14. What is a Kafka message?
    #+begin_quote
    A record consisting of a key, value, timestamp, and metadata
    stored in a partition.
    #+end_quote

15. How is Kafka configured to get acknowledgement of writes to only
    the leader?
    #+begin_quote
    Set the producer configuration `acks=1`.
    #+end_quote

16. What does the =ack=all= flag mean?
    #+begin_quote
    A write is acknowledged only when all in-sync replicas have
    confirmed the write.
    #+end_quote

17. Which Kafka configuration is needed for a consumer to read data
    from a topic?
    #+begin_quote
    The consumer needs bootstrap servers, a group ID, and subscription
    to the topic.
    #+end_quote

18. What is a Kafka client?
    #+begin_quote
    Any application using the Kafka API: producers, consumers, or
    admin tools.
    #+end_quote

19. Two consumers that have the same consumer group ID will read from
    mutually exclusive partitions. True or false?
    #+begin_quote
    True; partitions are divided among consumers within a group with
    no overlap.
    #+end_quote

20. Where are Kafka offsets stored?
    #+begin_quote
    In the internal Kafka topic ~__consumer_offsets~ (unless the
    consumer uses manual external storage).
    #+end_quote

** Review III: Kafka Architecture Review Sheet

*** Partitions and Log Structure
- Topics contain any number of partitions.
- Each partition is an append-only log.
- Offsets are per-partition and tracked by consumers.

*** Brokers and Replication
- A broker = Kafka server, identified by broker ID.
- Brokers host only some partitions.
- Replication factor determines how many brokers hold each partition.
- One leader per partition; remaining replicas follow.
- ISR = In-Sync Replicas fully caught up with the leader.

*** Producers
- Need bootstrap servers + target topic.
- Messages are immutable once written.
- acks=1 → leader-only acknowledgement.
- acks=all → all ISR replicas must acknowledge.

*** Consumers
- Need bootstrap servers + group ID + topic subscription.
- Offsets matter only to consumers (not producers).
- Offsets stored in __consumer_offsets unless overridden.
- Consumers in same group divide partitions: no overlap.

*** Fault Tolerance
- With replication factor N, system tolerates N-1 broker failures.
- Leader election among ISR ensures consistency.

*** Kafka Clients
- Any code using Kafka APIs: producer, consumer, admin, streams.
